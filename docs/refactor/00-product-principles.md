# 产品化重构原则

日期：2026-04-24

## 目标

这次重构的目标不是把代码拆得更漂亮，而是把 Subtitle Maker 变成一个稳定、可持续自用、后续可以产品化的工具。

所有工程决策都必须服务于一条主链路：

```text
上传视频
  -> 生成/上传 source.srt
  -> 智能分句
  -> 可选短句合并
  -> 翻译
  -> 一句一句配音
  -> Review 问题句
  -> 重配改动句
  -> 导出结果
```

## 从 13 条 SaaS 规则提炼出的本项目原则

| SaaS 规则 | 本项目落地 |
|---|---|
| Use your own product | 优先级来自真实使用中的失败、卡点、字幕质量和配音稳定性问题。 |
| Retention > acquisition | 先让当前核心用户稳定完成配音，不做拉新型功能。 |
| Cut your MVP in half | 第一阶段只保留字幕、翻译、配音、review、导出闭环。 |
| Landing page has 5 seconds | 页面必须让用户快速知道下一步该做什么，Agent V1 也要服务这个目标。 |
| Talk to your users | Agent V1 作为使用引导和问题收集入口，后续反向指导产品优化。 |
| Pay attention to market | 如果用户持续卡在某一步，优先修体验和稳定性，而不是继续堆架构。 |
| Think bigger | 架构给未来 SaaS 留口，但第一阶段不做账号、计费、多租户。 |

## 产品优先级

| 优先级 | 内容 | 原因 |
|---|---|---|
| P0 | 跑通并稳定主链路 | 没有稳定输出，架构和 UI 都没有价值。 |
| P0 | 错误可解释 | 当前失败常需要读日志，用户体验不稳定。 |
| P0 | 用户知道下一步 | 如果用户不知道该点哪里，功能再强也不可用。 |
| P1 | 字幕质量 | 后续翻译和配音质量都依赖 source.srt。 |
| P1 | 配音 review/redub 稳定 | 这是配音产品能否自用的关键。 |
| P1 | Agent 使用帮助 | 降低卡住时的排查成本。 |
| P2 | 模块拆分 | 为后续长期维护服务。 |
| P3 | SaaS 能力 | 账号、计费、团队协作暂不进入第一阶段。 |

## 产品成功指标

第一阶段不用商业指标衡量，而用自用闭环指标衡量。

| 指标 | 通过标准 |
|---|---|
| 首次操作清晰度 | 打开页面后，用户能在 5 秒内判断应该上传媒体、上传字幕或打开 Agent。 |
| 一分钟样例完成率 | 1 分钟视频能完成字幕、翻译、配音、review、导出。 |
| 失败可解释率 | DeepSeek、Index-TTS、ASR、ffmpeg 常见失败能显示可读原因和下一步建议。 |
| 字幕可用性 | `source.srt` 默认是一句一行倾向，必要时可开启短句合并提升配音效率。 |
| 重配安全性 | review redub 失败不能破坏已有音频和字幕产物。 |
| Agent 有效性 | Agent V1 能回答“下一步怎么做”和“这个错误怎么修”，但不声称已执行操作。 |

## 决策门槛

任何新增功能或重构任务都要先回答三个问题：

1. 是否直接提升主链路稳定性、字幕质量、配音质量或错误可解释性。
2. 是否会增加用户完成一次配音的步骤或理解成本。
3. 是否能通过测试、样例或日志证明没有破坏旧 API、旧 CLI、历史 batch。

如果答案不清楚，默认不进入第一阶段。

## 第一阶段不做

| 不做 | 原因 |
|---|---|
| Google Login | 当前是本地工具，账号体系不是核心瓶颈。 |
| 收费系统 | 还没稳定验证核心价值。 |
| 多租户 | 会显著增加数据隔离和部署复杂度。 |
| 团队协作 | 不是当前核心使用场景。 |
| 云端任务队列 | 先把本地任务状态和产物协议规范化。 |
| 自动说话人识别 | 当前主链路要求逐句参考情绪，不依赖 diarization/pyannote。 |
| 大规模前端框架迁移 | 先拆现有原生 JS，避免引入额外变量。 |

## 工程约束

- 所有重构必须先有文档和 review。
- 每个实现阶段必须有可验证的测试或手测证据。
- 不允许为了“架构正确”破坏旧 API、旧 batch、旧 CLI。
- 新模块先被旧入口调用，稳定后再清理旧实现。
- 出现行为差异时，先确认是 bug 修复还是回归，再继续推进。
